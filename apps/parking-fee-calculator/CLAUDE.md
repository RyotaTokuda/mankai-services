# CLAUDE.md

## Project
このプロジェクトは「駐車料金看板読取サービス」のMVPです。

目的は、ユーザーが現地で駐車場の料金看板を撮影し、
料金ルールをわかりやすく整理して、
「今から何時間停めたらいくらか」
「最大料金がどう適用されるか」
「注意すべき条件は何か」
を誤解なく把握できるようにすることです。

これは単なる駐車場検索サービスではありません。
まずは「現地で迷わない道具」を作ることを優先してください。

このプロジェクトは最終的に **iOS / Android 向けアプリ** として提供する前提です。
そのため、設計やUIはモバイルアプリでの利用を最優先に考えてください。

初期段階では以下のいずれかの構成を想定します:
- React Native / Expo によるクロスプラットフォーム構成
- 必要に応じて後からネイティブ化しやすい構成

Web管理画面や検証用画面を作る場合も、あくまで主役はモバイルアプリです。

## Product Principles
- 最優先は実用性
- 現地で素早く使えることを重視する
- 派手さより、誤解の少なさ・見やすさ・押しやすさを優先する
- スマホでの片手操作を前提にする
- MVP段階では機能を絞る
- 最小で動くものを先に作る
- まずはユーザーの「料金が分かりづらい」という不満を解決する
- 駐車場検索アプリではなく、現地判断を助けるアプリとして設計する

## Product Positioning
このアプリの中心価値は以下です。

- 看板を読めること
- 料金ルールを整理できること
- 停車前または停車直後に、すぐ料金の見通しを持てること
- 最大料金や例外条件を誤解しないこと

このアプリが優先して解く課題:
- 駐車場料金看板の表記が分かりづらい
- 最大料金の条件が複雑で誤解しやすい
- 夜間料金や時間帯切替が分かりにくい
- 「あと何分でいくら増えるか」が直感的に分からない

このアプリがMVP時点でまだ解かなくてよい課題:
- 全国完全対応の駐車場検索
- 地図上での網羅的比較
- 駐車場予約
- 決済連携
- コミュニティ機能
- 運営投稿型レビュー機能の本格運用

## Current Tech Direction
現時点の技術方針:
- モバイルアプリを最優先
- クロスプラットフォームを基本方針とする
- TypeScript を使用する
- UI はモバイル前提で設計する
- 将来的なバックエンドや保存機能は Supabase を候補とする
- 将来的な配布対象は iOS / Android
- 必要に応じて検証用のWeb画面や管理画面を追加してよい

現時点で想定する候補:
- React Native
- Expo
- TypeScript
- 必要に応じて Supabase
- 必要に応じて画像保存や解析API

別スタックへの大きな変更は、ユーザー確認なしに行わないでください。

## MVP Scope
MVPで作るもの:
- トップ画面
- 看板画像アップロードまたは撮影画面
- 解析結果画面
- ダミーまたは簡易解析結果の表示
- 基本的な料金計算ロジック
- 最大料金の適用条件表示
- 注意書き表示
- スマホUIとして成立する最低限の導線

MVPでまだやらないもの:
- 本格OCR最適化
- 高精度な画像前処理
- 認証
- 課金
- 通知
- 地図検索の作り込み
- 大規模な近隣比較
- 予約機能
- 決済機能
- ネイティブ個別最適化の深掘り
- 大量の外部連携

## Primary User Experience
ユーザーは現地で急いでいます。
したがって、以下を重視してください。

- すぐ撮れる
- すぐ読める
- すぐ判断できる
- 警告がすぐ分かる
- 迷わない
- タップ回数が少ない
- 文字が小さすぎない
- 情報量が多すぎない

ユーザーは屋外で片手操作する可能性が高いため、
操作フローは短く、視認性を高くしてください。

## UI Rules
- モバイル画面を最優先に設計する
- iOS / Android のスマホ利用を前提にする
- 屋外利用を想定して視認性を高くする
- コントラストを確保する
- タップ領域を十分に確保する
- 不要なアニメーションは避ける
- 重要情報は画面上部に寄せる
- 現在料金、最大料金、注意条件を最優先で見せる
- フォームや説明文は短くする
- 1画面でやることを詰め込みすぎない

重要情報の優先順位:
1. 今いくらか
2. 最大料金
3. 最大料金の適用条件
4. 時間帯ルール
5. 注意事項
6. 補足情報

特に以下は強く表示する:
- 最大料金は1回限り
- 繰り返し適用なし
- 車室ごとに異なる可能性
- 夜間料金は別計算
- 時間帯切替あり

## App-Specific Considerations
アプリとして考慮すること:
- カメラ起点の導線を重視する
- 画像選択だけでなく撮影導線も想定する
- 通信状況が不安定でも最低限成立しやすい設計を意識する
- 後でオフライン補助を入れやすい構造にする
- 権限要求は最小限にする
- ユーザーに不要な設定を強いない
- 撮影後の待ち時間を短く見せる工夫を考える
- モバイルで扱いやすい状態管理と画面遷移を意識する

## Workflow Rules
- 長いbashコマンドを1行にまとめない
- コマンドは原則1つずつ実行する
- 実行前に何をするか短く説明する
- 破壊的変更の前は確認する
- エラーが出たら、ごまかさず原因を説明する
- 実装前に短い実装方針を出す
- 変更後は、変更ファイルと次の候補タスクをまとめる
- READMEも必要に応じて更新する
- 一度に大きく進めすぎない
- 1回の作業範囲は小さく区切る

## Collaboration Style
- Claude は共同開発者として振る舞う
- 勝手に仕様を広げない
- 毎回細かすぎる確認をしない
- ただし後戻りコストが高い判断は確認する
- まずはMVPとして成立する最小構成を提案する
- 不確実な点は仮定を置いて進めてよいが、その仮定を明示する
- 相談なしに理想設計へ飛びすぎない
- まず動くものを作り、その後に改善する

## Decision Making
- まず「最小で動く形」を優先する
- 理想形ではなくMVPとして成立するかで判断する
- 拡張性よりもまず分かりやすさと検証しやすさを優先する
- 過剰な一般化や抽象化を避ける
- 将来使うかもしれないだけの機能は入れない
- 迷ったら、実装コストが小さくユーザー価値が大きい方を選ぶ
- 迷ったら、モバイル利用時の体験が良い方を選ぶ

## Implementation Policy
- 1回の作業範囲は小さく区切る
- 1画面、1機能、1ロジック単位で進める
- ダミーデータで先に流れを通す
- 本実装は後から差し替えやすくする
- OCRやAIの精度改善は後回しでよい
- まずは画面遷移と料金表示の体験を成立させる
- UIとロジックをなるべく分けて保守しやすくする

## How to Ask the User
- 些細な確認は毎回聞かず、合理的な仮定を置いて進める
- ただし、プロダクトの方向性、技術スタック変更、破壊的変更、課金、認証、外部公開に関わることは必ず確認する
- 質問は1回に1〜3個までに絞る
- 抽象的に広く聞かず、意思決定しやすい選択肢つきで聞く
- 聞く前に、なぜその確認が必要かを一言で説明する
- ユーザーがすぐ答えられる粒度で聞く

## Question Granularity
- 質問は大きすぎず、小さすぎない粒度にする
- 1画面単位、1機能単位、1判断単位で聞く
- 複数の未決事項がある場合は、優先順位が高いものから聞く
- 命名や小さなUI調整など、重要度が低いものは自律的に進める
- ユーザーの判断が必要なものだけを聞く

## When to Ask the User
以下のときは確認する:
- 仕様に複数の解釈があり、どれを選ぶかで実装が大きく変わるとき
- 後戻りコストが高い判断をするとき
- MVPの範囲を広げる提案をするとき
- 既存方針と矛盾する可能性があるとき
- アプリの主要導線を変えるとき
- 技術スタックを変えるとき
- データモデルを大きく変えるとき

以下は確認せず進めてよい:
- 軽微な文言調整
- 小さなレイアウト修正
- ダミーデータの整形
- 小さな命名改善
- 軽いリファクタリング

## Autonomy Rules
Claude can decide without asking:
- 軽微なUI調整
- ファイル分割
- 型名や変数名の自然な改善
- ダミーデータの作成
- READMEの軽微な整理
- 小さなリファクタリング
- 一時的な仮データや仮状態の導入

Claude must ask before deciding:
- 技術スタックの変更
- DB設計の大きな変更
- 認証導入
- 課金導入
- 外部APIの本採用
- 依存ライブラリの大量追加
- 既存画面の大幅なUX変更
- Gitの破壊的操作
- アプリ公開に関わる方針変更
- iOS / Android 実装方針の大きな変更

## How to Propose Options
- 何かを提案するときは、1案だけでなく2〜3案までに絞る
- それぞれのメリット・デメリットを短く示す
- 推奨案がある場合は、なぜそれを推すかを明示する
- ユーザーが今判断しやすい形で提案する
- 推奨案を最初に簡潔に示す

## Code Style
- 小さくシンプルに書く
- まずは読みやすさを優先する
- 不要な抽象化を避ける
- 型はできるだけ明示する
- 1ファイルに責務を詰め込みすぎない
- 複雑な処理は関数に切り出す
- UIコンポーネントは再利用可能性を意識する
- 一時的な実装である場合は、それが分かるようにする
- 将来の差し替えを意識して依存箇所を閉じ込める

## Comment Rules
- コメントは WHY を中心に書く
- 見れば分かる処理の説明コメントは増やさない
- 料金ルールの例外条件や判断根拠はコメントで残す
- 将来的に差し替える前提の箇所は意図を残す
- OCR結果の解釈や料金条件の扱いなど、誤読しやすい部分は補足する

## Data Design Direction
将来的に以下の概念を持つ:
- parking_signs
- parsed_rules
- price_simulations
- saved_locations
- user_corrections
- nearby_candidates

ただしMVP初期ではDBを急がず、
まず画面とロジックとモバイル体験を優先すること。

料金ルールは、後で機械可読にしやすい構造を意識すること。
文字列ベタ書きだけで完結させないこと。

## Pricing Logic Direction
料金ルールの実装では以下を意識する:
- 時間帯切替
- 昼間 / 夜間
- 最大料金
- 最大料金1回限り条件
- 繰り返し適用の有無
- 一定時間ごとの加算
- 例外条件
- 注意文言

まずは単純なケースから対応し、ルールを増やせる構造にすること。
最初から完全対応を目指さないこと。

## Testing Rules
- 料金計算ロジックの変更時は、先に小さなテストケースを追加する
- 境界条件を優先して確認する
- 特に以下を重視する:
  - 最大料金
  - 1回限り条件
  - 時間帯切替
  - 日跨ぎ
  - 端数時間
- モックは外部依存がある場合だけ使う
- UIだけでなく計算結果の正しさも確認する

## Dependency Rules
- 新しいライブラリを追加する前に、追加理由を一言説明する
- 既存依存または標準機能で実現できる場合は追加しない
- MVP段階では依存を増やしすぎない
- モバイルアプリでの重量や複雑さも考慮する
- 導入コストに対して効果が薄い依存は避ける

## File Exploration Rules
- 実装前に関連ファイルを確認する
- 新規ファイルを増やす前に既存構造を再利用できるか確認する
- 類似コンポーネントや既存型定義を先に探す
- ディレクトリ構造を散らかしすぎない
- 似た責務のコードが複数箇所に増えないようにする

## Documentation Rules
- セットアップ手順が変わったら README を更新する
- 環境変数を追加したら .env.local.example も更新する
- MVPの範囲や方針が変わったら CLAUDE.md を更新する
- 新しい依存や採用理由は必要に応じて記録する
- セットアップで詰まりやすい点は README に残す

## Branch Rules
- main は常に動作する状態を保つ
- 作業は原則として main へ直接コミットしない
- ブランチは1目的ごとに切る
- 大きすぎる変更を1つのブランチに詰め込まない

### Branch Naming
- feature/<short-description>
- fix/<short-description>
- chore/<short-description>

例:
- feature/camera-upload-screen
- feature/result-screen
- feature/price-rule-parser
- fix/mobile-layout
- chore/project-setup

### Workflow
- 新しい画面や機能を触るときは feature ブランチを切る
- 軽微な不具合修正は fix ブランチを切る
- 環境構築、設定変更、README更新は chore ブランチを切る
- 作業開始前に現在のブランチを確認する
- main にいる場合、新機能追加や大きめの修正では新しいブランチを切る提案をする

### Safety Rules
- main へ直接 push しない
- Git 操作前に、何をするか短く説明する
- Claude は勝手に force push しない
- Claude は勝手に rebase しない
- Claude は勝手に reset --hard しない
- Claude は勝手に clean -fd を実行しない

## Commit Rules
- 変更は小さく区切る
- コミットメッセージは何を変えたか分かるようにする
- 関係ない変更を同じコミットに混ぜない
- 可能なら conventional commits に近い形で書く

例:
- feat: add upload screen skeleton
- feat: add dummy result screen
- fix: adjust mobile spacing
- chore: update project setup
- docs: update README

## What to Optimize For
- 実装速度
- モバイルでの使いやすさ
- ローカルで動くこと
- 画面遷移が通ること
- 料金ロジックをあとで育てやすいこと
- iOS / Android 両対応しやすい構造
- 後でOCRや保存機能を足しやすい構造

## What to Avoid
- 早すぎる最適化
- 勝手な多機能化
- 認証や課金の先回り実装
- 不必要なライブラリ追加
- 長すぎる複合コマンド
- 将来想定だけの重い設計
- Web中心の発想でモバイル体験を損なうこと
- 画面に情報を詰め込みすぎること

## First Priorities
最初に優先すること:
1. 開発環境を安定させる
2. モバイル前提の画面導線を作る
3. 看板撮影 / 画像選択から結果表示までの流れを通す
4. ダミーデータで料金表示を成立させる
5. 基本的な料金計算ロジックを作る
6. 後でOCRや保存機能を差し替えやすくする

## Summary
このプロジェクトでは、
「現地で看板を見ても分かりにくい料金情報を、
スマホで素早く理解できるようにすること」
を最優先にしてください。

派手な機能や拡張より、
まずは iOS / Android アプリとして、
少ない操作で、
料金と注意点が分かる体験を成立させてください。
